CNTRLPLANE-3465

Hypershift Image Override for ARO HCP Testing

Enable testing PR-built HyperShift Operator and Control Plane Operator images against real ARO HCP environments before merge

Jira Ticket ARO-HCP PR #5467 CS MR !347 Release PR #79926

Problem

Today, when a developer opens a PR against the openshift/hypershift repository, there is no way to test their changes against a real ARO HCP environment before merge. The HyperShift Operator (HO) and Control Plane Operator (CPO) images are hardcoded in the ARO HCP deployment config and baked in at build time. This means hypershift changes that could break ARO HCP are only discovered after merge.

Solution

Make both the HO and CPO images overridable parameters, then wire up CI so that hypershift PR images are automatically built, pushed to Azure Container Registry (ACR), and deployed into a fresh ARO HCP environment provisioned from scratch for e2e testing.

This is a presubmit test on the hypershift repository that validates changes against a real ARO HCP deployment. It requires coordinated changes across three repositories.

Execution Flow: Developer Perspective

What happens when a developer opens a PR against openshift/hypershift:

Architecture: Each PR provisions a complete ARO HCP environment from scratch (regional infra, service cluster with all services including HO, management cluster with Maestro Agent and ACM), runs e2e tests, and tears everything down. This ensures complete isolation — concurrent PRs cannot interfere with each other. The workflow follows the existing aro-hcp-local-e2e pattern (provision → test → teardown). Test HostedCluster worker nodes are created in a dedicated Azure subscription (f464d333-13e0-40c6-a3de-a8e8cb32c939). Only global infrastructure (ACR, DNS parent zones) is shared.

Why Provision Per-PR?

Concurrency Isolation

The HyperShift Operator is a cluster-wide singleton deployment on the service cluster. Concurrent PRs sharing the same service cluster would cause serious conflicts:

  • Image conflicts: If PR-A deploys HO image A, then PR-B deploys HO image B, PR-A's tests are now running against the wrong HO version
  • State corruption: HO manages cluster-wide resources (CustomResourceDefinitions, webhooks, RBAC) that would be overwritten by concurrent deployments
  • Test interference: HostedClusters created by different PRs would all be managed by whichever HO was deployed most recently

Solution: Each PR provisions its own complete environment (regional infra, service cluster, management cluster). This ensures:

  • Complete isolation — PRs cannot interfere with each other
  • Parallel execution — multiple PRs can test simultaneously
  • Clean state — each test starts from a known-good baseline

Merge Order & Dependencies

The three changes are independent at merge time but all must be merged before the e2e flow works end-to-end.

ChangeCan merge independently?Effect if merged alone
PR #5467 ARO-HCP Yes No-op - hoImageOverride and cpoImageOverride default to empty string, so existing behavior is unchanged
MR !347 Cluster Service Yes No-op - if no control_plane_operator_image property is set, no annotation is added
PR #79926 openshift/release Should merge last The e2e-aro-hcp test job exists but will fail if ARO-HCP and CS changes aren't merged and deployed
Recommended merge order: ARO-HCP and CS can merge in any order (both are no-ops when overrides aren't set). openshift/release should merge after both are deployed (they deploy automatically post-merge via ARO-HCP's CI pipelines).

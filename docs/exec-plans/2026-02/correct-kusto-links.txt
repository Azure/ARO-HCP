# Custom-Link-Tools: Config-Driven Environment Info

## Context

The custom-link-tools generates HTML pages with Kusto Data Explorer links for prow CI test runs. Currently it:
- Requires steps.yaml to derive the service cluster name (by parsing ARM operations) and earliest start time -- if the file is missing, the tool fails
- Hardcodes the Kusto Data Explorer URL to hcp-dev-us-2.eastus2
- Uses the service cluster name for all queries, including Hypershift and ACM which actually run on management clusters

This change makes the tool config-driven, more resilient, and correct.

### 2. Add config loading, kusto region logic, and new option fields

Config loading -- reuse pattern from tooling/helmtest/internal/config.go:26-39:
  import configtypes "github.com/Azure/ARO-Tools/config/types"
Load with yaml.Unmarshal into configtypes.Configuration, access values via cfg.GetByPath().

New struct: KustoInfo with KustoName, KustoRegion, ServiceLogsDatabase, HostedControlPlaneLogsDatabase

New fields in completedOptions: SvcClusterName, MgmtClusterName, Kusto KustoInfo

Kusto region resolution -- hardcoded mapping derived from:
- dev-infrastructure/kusto-pipeline.yaml execution constraints
- ARO-Tools/pkg/config/ev2config/config.yaml geoShortId->region mapping

Dev environments: all use hcp-dev-us-2 in eastus2. Detect via strings.HasPrefix(kustoName, "hcp-dev-").

Public cloud: kusto names follow the format hcp-<env>-<geoShortId>[optional-suffix] where the suffix
may be digits appended directly (e.g. hcp-prod-ch2) or hyphen-separated text (e.g. hcp-stg-br-5,
hcp-stg-br-second-ateempt-7). Extract geoShortId as the first 2 characters after hcp-<env>- and
look up in map:
  au -> australiaeast
  br -> brazilsouth
  ca -> canadacentral
  ch -> switzerlandnorth
  eu -> westeurope
  in -> centralindia
  uk -> uksouth
  us -> eastus2

# CI Documentation Maintenance Guide

This file contains notes for maintaining the CI documentation subtree under `docs/ci/`.

## Scope

The canonical CI docs now live in:

- `docs/ci/README.md`
- `docs/ci/execution.md`
- `docs/ci/image-lifecycle.md`
- `docs/ci/identity-leasing.md`
- `docs/ci/ev2-integration.md`
- `docs/ci/cleanup.md`
- `docs/ci/e2e-testing.md`
- `docs/ci/operations.md`

The old top-level CI docs were intentionally removed. Do not recreate the old large `prow.md` job catalog.

## Documentation Shape

Keep the CI docs concept-first:

- `README.md`: where to start and which doc answers which question
- `execution.md`: how CI works and what each mode validates
- `image-lifecycle.md`: deep mechanics for the CI image graph, promotion, and ACR push boundary
- `identity-leasing.md`: deep mechanics for managed identity and MSI mock SP leasing
- `ev2-integration.md`: deep mechanics for EV2, Gangway, and pinned gating runs
- `cleanup.md`: why cleanup paths differ
- `e2e-testing.md`: how contributors trigger or narrow tests
- `operations.md`: how maintainers trigger, inspect, troubleshoot, or change CI

The only place that should keep even a small job inventory is the tiny appendix in `docs/ci/operations.md`.

Do not reintroduce:

- large per-job tables
- schedule dumps in the main narrative
- duplicated lists of job names across multiple files

## Which File To Update

Update `docs/ci/execution.md` when:

- the CI execution modes change
- the Azure request path or tenant split changes
- EV2 commit pinning or `prowJobName` wiring changes
- the identity-leasing runtime model changes

Update `docs/ci/image-lifecycle.md` when:

- the shared CI build root or test runner image lifecycle changes
- the job-local image graph changes in `openshift/release`
- local E2E stops injecting freshly built service images into the environment, or that flow changes shape
- ci-operator promotion rules or the `images-push` boundary change
- the log-reading mental model needs to reflect a different dependency graph

Update `docs/ci/identity-leasing.md` when:

- the ci-operator lease wiring changes
- managed identity pool sizing or quota assumptions change
- MSI mock SP pool setup or lookup behavior changes
- Boskos resource types or counts for identity-related pools change

Update `docs/ci/ev2-integration.md` when:

- EV2 validation steps or Gangway auth change
- `prowJobName` mapping or `gatePromotion` wiring changes
- commit-pinning behavior changes
- rollout metadata or correlation annotations change

Update `docs/ci/cleanup.md` when:

- cleanup semantics change
- a cleanup path changes owner or strictness
- the location of cleanup implementation changes

Update `docs/ci/e2e-testing.md` when:

- `/test` commands change
- contributor guidance for higher-environment testing changes
- test filtering guidance changes

Update `docs/ci/operations.md` when:

- how to inspect or correlate runs changes
- CI config maintenance workflow changes
- pool-management procedures change
- the small appendix needs a new job family or a removed one

Update `docs/ci/README.md` when:

- the doc set changes shape
- the recommended reader journey changes

## Update Workflow

When CI config changes in `openshift/release`:

1. update the relevant source config there
2. follow the release repo's documented regeneration flow (`make update`)
3. update the ARO HCP docs only if the change affects behavior, navigation, or the small appendix

When the change is only a job rename or schedule tweak, prefer updating the appendix in `docs/ci/operations.md` rather than expanding the conceptual docs.
